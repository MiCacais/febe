# 🤝 FeBe Take-Home Assessment Agreement & Contract

**Purpose:** This document is a bilateral agreement to be completed, agreed upon, and signed by both the hiring manager and the candidate before any take-home work begins. 

FeBe mandates that all take-home assessments must be fully compensated to respect the candidate's professional time and enforce pipeline discipline on the hiring organization. We believe that being compensated for labor brings a much better signal to the company regarding performance, as candidates deliver their absolute best work when financially incentivized.

---

## 🛑 The 5 Non-Negotiable Rules of FeBe Take-Homes

1. **Strict 2-to-3-Hour Cap:** The scope of the exercise must be intentionally designed to be fully completed within a maximum of 3 hours by an engineer of the target seniority level. If a candidate must spend a weekend on it, the assignment is non-compliant.
2. **Guaranteed Financial Compensation:** The candidate **must** be compensated for their implementation time, regardless of whether they advance to the final round. Free labor is explicitly banned.
3. **Flat-Rate Compensation Value:** A strict flat fee of **$150 to $200 USD** (or local currency equivalent) must be determined, agreed upon, and locked in prior to the start of work.
4. **Zero Exploitation of Labor:** The assignment must use a mocked domain or fictional problem space. Under no circumstances can a company ask a candidate to build features, fix bugs, or write code that will be used in the company’s actual production codebase.
5. **No Hidden Grading Criteria:** The evaluation rubric must be shared with the candidate inside this agreement so they know exactly how their code will be analyzed.

* **Open Source Freedom:** The candidate is fully permitted to host their completed assignment in a public or private personal GitHub repository to showcase as part of their professional portfolio.

## 📊 The Evaluation Rubric (Open & Transparent)

Your submission will be evaluated strictly on engineering execution, not architectural perfection art. Our team will grade your code based on the following three binary pillars:

| Evaluation Pillar | Green Flags (Pass) | Red Flags (Fail) |
| :--- | :--- | :--- |
| **Correctness & Scope** | The code successfully executes the core baseline requirements specified in the prompt within the requested time box. | The application fails to boot, crashes on baseline data, or the candidate over-engineered features outside the prompt. |
| **Readability & Structure** | Variables are named intentionally, functions are decoupled logically, and a peer engineer can easily navigate the codebase. | Monolithic code walls, chaotic layout, or a complete lack of structural separation that makes code unmaintainable. |
| **Pragmatic Production Discipline** | Includes basic error handling for obvious failure states and includes a brief `README.md` explaining trade-offs made due to the 3-hour limit. | Code completely panics on minor edge cases, or assumes a flawless environment with zero error boundaries. |
